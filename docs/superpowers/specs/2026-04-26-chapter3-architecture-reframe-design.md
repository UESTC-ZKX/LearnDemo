# 第三章剩余模块页面重构设计

日期：2026-04-26
主题：第三章 Claude 工程实现解析页面重构

## 背景

当前第三章已经具备一张全局架构图，但剩余 5 个关键模块仍沿用普通 stage card 的展示方式：

- 图片与讲解文本没有形成稳定的视觉结构
- 5 张图之间缺少明确的前后承接关系
- “讲解重点”以段落为主，不利于现场演讲快速扫读
- 页面虽然有内容，但没有形成一条清晰的工程叙事链

这次重构的目标不是继续堆图片，而是把第三章改造成一组高可视化的“架构解读卡”，让听众能够顺着固定顺序理解：

`思考 -> 行动 -> 运行时 -> 协作 -> 扩展`

## 目标

1. 保留第三章“从工程模块角度讲 Claude 系统”的定位。
2. 让全局架构图和 5 张关键模块图形成一条完整的讲解路径。
3. 用结构化视觉区域替代普通图文堆叠，提高演讲时的可读性和可扫读性。
4. 尽量复用当前 Chapter/Stage 数据结构与滚动式章节体验，避免整页推翻重做。
5. 为后续继续精修样式、补充交互和局部动画留下清晰骨架。

## 非目标

- 不修改前三章的整体信息架构。
- 不把第三章做成独立路由或完全独立页面。
- 不在本轮加入复杂数据驱动高亮、图内热点映射或重型浏览器交互。
- 不重写用户提供的 5 张图内容本身，先以现有图片作为核心视觉资产。

## 推荐方案

采用“总览锚点 + 纵向模块卡”方案。

页面结构分为两层：

1. 顶部总览层
   - 保留已有全局架构图
   - 在总图下增加 5 步模块导航链
   - 用一句话说明第三章主线：思考、行动、运行时、协作、扩展如何连成系统

2. 模块解读层
   - 将 5 个模块按固定顺序垂直排布
   - 每个模块都采用统一的“架构解读卡”结构
   - 卡片之间用过渡句和前后关系条连接，形成滚动讲解链

这样既保留现有项目的长页面讲述体验，也显著提升第三章的视觉组织力。

## 页面结构设计

### 1. 顶部总览区

用途：先让用户看见全局，再决定接下来 5 张图分别在全局中的位置。

组成：

- 第三章标题与总述
- 已有全局架构图
- 5 步导航链
  - Conversation Engine
  - Tool System
  - ToolUseContext + AppState
  - Task / Subagent
  - Extension Layer
- 一句总过渡文案
  - 例如：Agent 先完成一轮思考，再触发行动，依托运行时骨架扩展成协作系统，最终演化为可持续扩展的平台。

交互：

- 点击导航链可跳转到对应模块卡
- 当前滚动区域对应的模块在导航链中高亮

### 2. 架构解读卡

每个模块统一使用四段式结构，避免每块重新发明布局。

#### A. 主图可视区

- 放大展示用户提供的模块图
- 图片容器保持高对比、强留白和稳定比例
- 移动端采用上下布局，桌面端采用左图右文或上图下文的稳定模式

#### B. 讲解重点区

- 将用户提供的“讲解重点”提炼为 3-5 个高亮点
- 使用信息标签、重点条、数字步骤或对照卡表达
- 不再展示成长段描述

示例：

- Conversation Engine：7 步主循环做成竖向步骤带
- Tool System：并发安全 / 非并发安全做成左右对照面板
- Runtime：ToolUseContext / AppState 做成双核说明块
- Task / Subagent：状态机与执行单元做成分层标签区
- Extension：统一抽象做成能力映射带

#### C. 一句话概括区

- 将每张图的“你讲这张图时可以一句话概括”提升成显眼收束句
- 视觉上与正文区分，作为该模块的记忆钉子

#### D. 前后关系区

- 明确该模块在主线中的位置
- 说明“它承接上一张图什么问题，又把什么问题交给下一张图”

示例：

- Conversation Engine -> Tool System
  - 先定义一轮思考怎么跑完，再解释思考中产生的动作如何被系统安全执行
- Tool System -> Runtime
  - 工具能执行之后，接下来要回答这些执行能力依托什么共享运行时存在

## 5 个模块的内容落位

### 模块 1：Conversation Engine / QueryEngine

定位：

- 第三章的第一张关键模块图
- 回答“Agent 为什么不是一次普通模型调用”

页面重点：

- 强调输入来源不是只有用户一句话
- 强调中间 7 步主循环是核心结构
- 强调右侧是运行时能力，而不是普通配置
- 底部横向流程用来做总结

推荐表达：

- 左侧重点标签：输入来源、上下文组装
- 中部竖向步骤带：7 步主循环
- 右侧能力胶囊：Streaming / Retry / Budget / Abort
- 收束句：它定义了 Agent 一轮完整思考是怎么跑完的

### 模块 2：Tool System + Tool Orchestration

定位：

- 回答“Agent 为什么能行动，而且行动不是混乱的”

页面重点：

- 工具来源多路接入
- Tool Registry 是统一登记层
- Tool Orchestrator 的价值是调度而不只是执行
- 并发安全 / 非并发安全是工程治理关键

推荐表达：

- 顶部结构条：工具来源 -> Tool Registry -> Orchestrator -> 输出
- 中部重点条：接收、校验、权限、并发、执行、收集、汇总、回写
- 底部双栏：并发安全 vs 非并发安全
- 收束句：它把“模型想做什么”变成“系统如何安全、可控地去做”

### 模块 3：ToolUseContext + AppState

定位：

- 回答“Agent 的运行时骨架是什么”

页面重点：

- Runtime Context Hub 是总中心
- ToolUseContext 代表工具执行现场
- AppState 代表全局状态树
- 会话、任务、UI、插件共享同一套运行时事实

推荐表达：

- 中央双核卡：ToolUseContext / AppState
- 两侧能力条：现场上下文能力、全局状态能力
- 底部消费者区：Conversation Engine / Tool Execution / UI
- 收束句：它让工具、会话、任务、界面共享同一套运行时事实

### 模块 4：Task / Subagent / Multi-Agent System

定位：

- 回答“Agent 如何从单体助手变成协作系统”

页面重点：

- 主 Agent 不直接做所有事
- 调度器负责拆解、依赖、并发和重试
- Task Store 让任务变成可持久、可恢复的单元
- 状态机要突出 pending -> running -> blocked -> completed / failed / cancelled

推荐表达：

- 左中右主链：主 Agent -> 调度器 / 任务存储 -> 执行单元
- 右侧总结栏：协作能力
- 底部大状态机区
- 收束句：它让 Agent 不再只是单线程对话助手，而是可拆解、可协作、可长期运行的系统

### 模块 5：Extension Layer

定位：

- 回答“Agent 为什么是平台，而不是写死的程序”

页面重点：

- 扩展来源分为 Skills、Plugins、MCP / Protocols
- Extension Adapter Layer 是能力吸收层
- Runtime Integration 是统一注册和治理层
- 所有扩展都必须映射到统一抽象

推荐表达：

- 三列主结构：扩展来源 -> 适配层 -> 运行时集成
- 中下方用强调带突出统一抽象
- 抽象标签：Command / Tool / Task / Prompt / Resource
- 收束句：它让外部能力接入后仍然受统一治理，而不是把系统越扩越乱

## 数据与组件设计

### 数据层

当前 `chapter-claude` 已存在 stage 数据，可继续复用其顺序和基本字段。

建议增加第三章专用展示配置，而不是强行复用普通 stage 文案结构：

- 模块编号
- 模块短标签
- 模块长标题
- 图片路径
- 一句话概括
- 核心讲解点数组
- 前后承接文案
- 强调标签数组

这样可以保留 `chapters.ts` 作为内容入口，同时避免在组件内写死大量展示文案。

### 组件层

建议引入或拆分以下组件：

- `ClaudeArchitectureChapterSection`
  - 第三章整体容器
- `ArchitectureModuleNav`
  - 顶部 5 步导航链
- `ArchitectureModuleCard`
  - 单个模块解读卡
- `ArchitectureKeyPointList`
  - 结构化讲解重点区
- `ArchitectureHandoff`
  - 模块间承接条

`ChapterSection` 继续作为章节分发入口，但第三章应像第二章一样拥有自己的专用 section 组件，而不是继续走通用 `StageCard`。

## 视觉设计方向

### 整体风格

- 沿用第三章当前深色工程感底色
- 用不同模块色作为节奏提示，但保持统一框架
- 总览层偏“总控台”
- 模块层偏“技术展板”

### 视觉原则

- 图片必须足够大，优先保证图可看清
- 文本不和图抢视觉中心
- 每卡只突出 1 个核心问题
- 模块之间通过箭头、编号、流程词强化顺序感

### 响应式策略

- 桌面端：导航链横向展开，模块卡采用双列布局
- 平板端：导航链可换行，卡片局部收缩
- 手机端：总览图、模块图、重点区改为单列堆叠

## 交互设计

### 必须有的轻交互

- 点击顶部模块导航滚动到对应区域
- 当前阅读模块高亮
- 模块进入视窗时出现轻量动效

### 本轮不做的重交互

- 图内热点点击
- 随滚动动态切换图片局部放大
- 多层弹出解释器

## 错误处理与降级策略

- 若部分图片资源尺寸不一致，卡片容器统一做自适应边框与留白，不裁切关键信息
- 若个别模块文案过长，优先压缩为结构化条目，不让卡片失衡
- 若移动端空间不足，优先保图，其次减少辅助标签数量

## 测试策略

本轮实现后至少覆盖以下验证：

1. 第三章仍能正常渲染总览图与 5 个模块
2. 模块导航点击能定位到正确 section
3. 一句话概括和讲解重点均可见
4. 移动端布局不会出现严重溢出
5. 现有章节切换与演讲模式不回归

测试形式：

- 组件渲染测试
- 关键文案可见性测试
- 导航交互测试

## 实施边界

本次实现聚焦在第三章剩余内容重构，不延伸到：

- 第二章 Agent 选型页重构
- 第一章模型演进页联动调整
- 新增额外 demo 引擎
- 图像内容二次生成或重绘

## 验收标准

满足以下条件视为本轮完成：

1. 第三章从“总图 + 普通 stage 卡片”升级为“总览 + 5 个架构解读卡”
2. 5 个模块顺序与叙事链清晰
3. 每个模块都可直接支持演讲讲解
4. 页面明显比当前版本更可视化、更便于扫读
5. 不破坏现有项目整体导航和章节结构

## 结论

第三章最合适的重构方式，不是再增加更多说明文字，而是把用户给出的 5 张图收编到统一的讲解框架中。页面应先用全局架构图建立全局地图，再用 5 个高一致性的模块卡完成工程叙事，让听众按“思考 -> 行动 -> 运行时 -> 协作 -> 扩展”理解 Claude 类系统的组织方式。
